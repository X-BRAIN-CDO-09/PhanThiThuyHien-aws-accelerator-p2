01
Why Orchestration

Tại sao cần Kubernetes?

Container giải quyết "chạy được trên máy tôi". Còn lại thì sao?

The Problem

Container thì dễ — chạy 200 container mới khó

Một docker run thì ai cũng làm được. Production cần nhiều hơn thế.

Docker cho bạn

  • Đóng gói app + dependency thành 1 image bất biến
  • Chạy giống nhau ở mọi nơi (laptop, CI, server)
  • Cô lập process, nhẹ hơn VM nhiều

Docker KHÔNG cho bạn

  • Container chết lúc 3h sáng → ai khởi động lại?
  • Traffic tăng 5x → ai scale thêm replica?
  • Deploy version mới → rollout an toàn, rollback ra sao?
  • 10 container trên 5 máy → ai quyết định đặt cái nào ở đâu?
  • Container A cần gọi B → tìm địa chỉ B kiểu gì khi IP đổi liên tục?

Kubernetes = orchestrator. Bạn mô tả trạng thái mong muốn ("tôi muốn 3 bản app này luôn chạy, expose ra cổng 80"), K8s lo phần làm sao để đạt và giữ trạng thái đó — kể cả khi máy chết, traffic tăng, hay bạn deploy version mới.

Mental Model

Tư tưởng cốt lõi: Desired State

Khác biệt lớn nhất so với "tự chạy script" — đây là vòng lặp điều khiển (control loop)

Declare
Bạn khai báo state mong muốn (YAML)
→

→
Diff
So sánh: thực tế vs mong muốn
→
Reconcile
Hành động để khớp lại

Imperative (cách cũ)

"Chạy container này, rồi cái kia, nếu chết thì... tôi tự xử." Bạn ra lệnh từng bước. Máy chết = bạn phải biết và sửa tay.

Declarative (cách K8s)

"Tôi muốn luôn có 3 replica." K8s liên tục kiểm tra và tự kéo về 3 nếu thiếu. Bạn mô tả cái gì, không phải làm thế nào.

02
Architecture

Một cụm K8s cấu tạo ra sao?

Control plane (bộ não) + Worker nodes (cơ bắp)

Big Picture

Control Plane vs Worker Nodes

Control Plane — bộ não

Ra quyết định cho toàn cụm: lập lịch, phản ứng sự kiện, giữ desired state. Không chạy app của bạn (thường thế).

  • kube-apiserver — cửa ngõ duy nhất, mọi thứ đi qua đây
  • etcd — database lưu toàn bộ state cụm
  • scheduler — quyết định Pod chạy trên node nào
  • controller-manager — chạy các control loop

Worker Node — cơ bắp

Nơi container của bạn thực sự chạy. Mỗi node là 1 máy (VM hoặc bare-metal).

  • kubelet — agent trên node, nhận lệnh từ apiserver và chạy Pod
  • container runtime — containerd / CRI-O, thực sự chạy container
  • kube-proxy — lo network routing cho Service trên node

Các công cụ chạy cụm local (minikube / kind) gom cả control plane và worker vào một node duy nhất — đủ để học và demo. Production thì control plane tách riêng, nhiều worker node. Lát nữa trong lab bạn sẽ dựng đúng loại cụm 1-node này.

4 thành phần, mỗi cái 1 việc

ComponentNhiệm vụVí dụ cụ thể
kube-apiserverREST API trung tâm. kubectl, kubelet, controller đều nói chuyện qua đây. Auth, validate, ghi vào etcd.Bạn gõ kubectl apply → request tới apiserver
etcdKey-value store nhất quán (Raft). "Nguồn sự thật" duy nhất về state cụm."Có Deployment X, muốn 3 replica" được lưu ở đây
schedulerPod chưa gán node? Scheduler chọn node phù hợp (đủ CPU/RAM, thoả ràng buộc).Pod mới → đặt lên node ít tải nhất
controller-managerChạy hàng loạt control loop. Đây là "reconcile" — kéo thực tế về mong muốn.Pod chết → ReplicaSet controller tạo Pod mới

Quy tắc vàng: không ai ghi thẳng vào etcd. Mọi thay đổi đi qua apiserver. Đây là lý do API server là điểm trung tâm cần bảo vệ — và là endpoint mà mọi client (kể cả tool tự động) kết nối tới.

02 — Architecture
End to End

Điều gì xảy ra khi bạn kubectl apply một Deployment?

# 1. kubectl gửi YAML tới apiserver (auth + validate) kubectl apply -f deploy.yaml ↓ # 2. apiserver ghi object Deployment vào etcd # 3. Deployment controller thấy có Deployment mới → tạo ReplicaSet # 4. ReplicaSet controller thấy cần 3 Pod, hiện 0 → tạo 3 Pod (pending) # 5. scheduler thấy 3 Pod chưa có node → gán mỗi Pod 1 node # 6. kubelet trên node nhận lệnh → gọi container runtime kéo image + chạy # 7. Pod Running. Mỗi bước đều ghi state ngược lại etcd qua apiserver

Không có "trung tâm điều phối" đọc kịch bản. Mỗi controller chỉ làm một việc nhỏ và phản ứng với thay đổi state. Ghép lại = hành vi tự phục hồi mạnh mẽ. Đây là tinh thần "level-triggered, không phải edge-triggered".

Core Objects

Những đối tượng bạn sẽ dùng mỗi ngày

Pod, Deployment, Namespace, Config — nền của mọi workload

Object #1

Pod — đơn vị nhỏ nhất K8s quản lý

Không phải container. Là nhóm 1+ container chạy chung, share network + storage.

apiVersion: v1 kind: Pod metadata: name: web labels: app: web spec: containers: - name: nginx image: nginx:1.27 ports: - containerPort: 80

Đặc điểm chính

  • Mọi container trong Pod share 1 IP + cùng localhost
  • Pod là ephemeral — chết là mất, IP đổi. Đừng bám vào IP của Pod.
  • Bạn hầu như không tạo Pod trực tiếp — để Deployment làm.

Vì sao "nhóm" chứ không phải 1 container?

Sidecar pattern: container chính + 1 container phụ (log shipper, proxy) cần chạy sát nhau, share volume/network. Pod là ranh giới đó.

03 — Core Objects
Object #2

Deployment — cái bạn dùng 90% thời gian

Quản lý ReplicaSet, lo scaling + rolling update + rollback. App của bạn sẽ deploy qua đây.

apiVersion: apps/v1 kind: Deployment metadata: name: web spec: replicas: 3 # desired state selector: matchLabels: app: web template: # khuôn để tạo Pod metadata: labels: {app: web} spec: containers: - name: web image: myapp:v1

Deployment → ReplicaSet → Pod

Deployment tạo ReplicaSet, ReplicaSet giữ đúng số Pod. Đổi image → Deployment tạo ReplicaSet mới, dịch dần traffic sang (rolling update).

Scale tức thì

kubectl scale deploy/web --replicas=5 hoặc sửa YAML rồi apply. K8s lo phần còn lại.

Self-healing

Kill 1 Pod thử xem — vài giây sau có Pod mới thay thế. Lát nữa bạn sẽ tự làm điều này trong lab.

03 — Core Objects
Object #3 & #4

Namespace & Labels — cách K8s "phân loại"

Namespace — vách ngăn logic

Chia cụm thành vùng tách biệt (dev / staging / prod, hoặc theo team). Tên object chỉ cần unique trong namespace.

  • default — nơi mọi thứ rơi vào nếu không khai báo
  • kube-system — component của chính K8s
  • kubectl get pods -n <ns> để xem theo namespace

Labels & Selectors — chất keo

Cặp key=value gắn lên object. Đây là cách mọi thứ "tìm" nhau — không phải qua IP.

  • Pod gắn app=web
  • Service dùng selector app=web → tự tìm đúng Pod
  • Deployment dùng selector để biết Pod nào thuộc về mình

Đây là ý tưởng cực quan trọng: liên kết trong K8s là loose coupling qua label, không phải hard-wire IP. Pod chết và sinh lại với IP mới — nhưng cứ giữ đúng label thì các object khác vẫn tìm thấy nó.

03 — Core Objects
Object #5 & #6

ConfigMap & Secret — tách config khỏi image

12-factor: cùng 1 image chạy mọi môi trường, chỉ đổi config bên ngoài

ConfigMap

Lưu config không nhạy cảm: URL, feature flag, tên môi trường. Inject vào Pod qua env var hoặc file mount.

kind: ConfigMap data: APP_ENV: "production" API_URL: "http://api:8080"

Secret

Giống ConfigMap nhưng cho dữ liệu nhạy cảm: mật khẩu, token, key. Mã hoá base64 (chú ý: không phải mã hoá thật — cần bật encryption-at-rest cho etcd).

kind: Secret type: Opaque data: DB_PASSWORD: cGFzc3dvcmQ= # base64

Hands-on Lab

Gấp slide lại — mở terminal lên

Tự tay tạo từng object vừa học. Mỗi slide là một bước — gõ theo, đợi output, rồi sang slide sau.

Reference

kubectl — thẻ tra cứu trước khi gõ

Giữ slide này mở bên cạnh trong suốt lab

# XEM kubectl get pods -A # mọi pod, mọi ns kubectl get deploy,rs,pods # phả hệ kubectl describe pod <name> # chi tiết + Events kubectl logs -f <pod> # log live kubectl get pods --show-labels
# LÀM kubectl apply -f app.yaml # declarative kubectl run NAME --image=... # 1 pod nhanh kubectl scale deploy/web --replicas=5 kubectl exec -it <pod> -- sh # vào trong pod kubectl delete -f app.yaml

Mẹo debug xuyên suốt: Pod không chạy? Theo thứ tự get → describe (xem Events ở cuối) → logs. 90% lỗi (ImagePullBackOff, CrashLoopBackOff, Pending) lộ ra ở 3 bước này.

Lab 0 · Cài đặt

Chọn 1 công cụ — minikube hoặc kind

Cả hai cho bạn 1 cụm K8s thật trên máy. Chọn 1, dùng xuyên suốt cả ngày.

Phương án A — minikube

# macOS brew install minikube # Linux curl -LO https://storage.googleapis.com/\ minikube/releases/latest/minikube-linux-amd64 sudo install minikube-linux-amd64 \ /usr/local/bin/minikube # Windows (choco) choco install minikube

Có nhiều addon sẵn (dashboard, ingress, metrics-server).

Phương án B — kind

# macOS brew install kind # Linux curl -Lo ./kind https://kind.sigs.k8s.io/dl/\ latest/kind-linux-amd64 chmod +x kind && sudo mv kind /usr/local/bin/ # Windows (choco) choco install kind

Rất nhẹ & nhanh, cụm = container Docker.

👉 Điều kiện chung: cần kubectl (mac brew install kubectl · Linux qua curl · Windows choco install kubernetes-cli), Docker đang chạy (docker ps không lỗi), và kubectl version --client in ra version. Windows: bật WSL2 + Docker Desktop. ✔ Xong thì sang Lab 1.

Lab 2 · Pod trần

Tạo 1 Pod, vào trong nó, rồi xoá

👉 Gõ theo
kubectl run hello \ --image=nginx:1.27 --port=80 kubectl get pods -o wide # thấy IP + node kubectl describe pod hello # đọc Events # Vào trong Pod kubectl exec -it hello -- sh -c \ 'hostname; nginx -v' # Xoá Pod trần kubectl delete pod hello kubectl get pods # → trống!
✅ Bài học

Pod là ephemeral

Xoá Pod trần là mất hẳn — không có gì tạo lại. Không ai trông nó cả.

Đây là lý do tồn tại Deployment: bạn gần như không bao giờ chạy Pod trần ở production. Lab 3 sẽ cho thấy khác biệt.

Lab 3 · Deployment (YAML)

Khai báo bằng file & apply

👉 Tạo file rồi apply
cat <<'EOF' > web.yaml apiVersion: apps/v1 kind: Deployment metadata: {name: web} spec: replicas: 3 selector: {matchLabels: {app: web}} template: metadata: {labels: {app: web}} spec: containers: - {name: web, image: nginx:1.27} EOF kubectl apply -f web.yaml kubectl get deploy,rs,pods
✅ Output mong đợi
deployment.apps/web 3/3 replicaset.apps/web-5d8c 3 pod/web-5d8c-aa Running pod/web-5d8c-bb Running pod/web-5d8c-cc Running
Declarative: web.yaml là nguồn sự thật. Muốn đổi gì → sửa file → apply lại. Thấy đúng phả hệ Deploy→RS→Pod như slide lý thuyết.

Lab 4 · Labels & Self-heal

Lọc theo label & "giết" 1 Pod

👉 Gõ theo
# Label — chất keo kubectl get pods --show-labels kubectl get pods -l app=web # lọc kubectl logs -l app=web --tail=3 # Self-healing: xoá 1 pod của Deployment kubectl delete pod \ $(kubectl get pod -l app=web \ -o name | head -1) kubectl get pods -w # pod mới mọc lên ngay
✅ So sánh với Lab 1

Cùng thao tác xoá — kết quả khác hẳn

Pod trần (Lab 2) xoá là mất. Pod thuộc Deployment xoá thì sống lại ngay — vì ReplicaSet giữ desired state = 3.

Control loop chạy trước mắt bạn: thực tế (2) ≠ mong muốn (3) → reconcile → tạo lại. Không ai bấm nút gì cả.
04 — Hands-on Lab
Lab 5 · Config & Secret

Tách config khỏi image, inject qua env

👉 Gõ theo
# Tạo ConfigMap + Secret kubectl create configmap app-cfg \ --from-literal=APP_ENV=production kubectl create secret generic app-sec \ --from-literal=DB_PASSWORD=s3cr3t # Inject vào Deployment qua env kubectl set env deploy/web \ --from=configmap/app-cfg kubectl set env deploy/web \ --from=secret/app-sec # Kiểm tra ngay trong Pod kubectl exec deploy/web -- env \ | grep -E 'APP_ENV|DB_PASSWORD'
✅ Output mong đợi
APP_ENV=production DB_PASSWORD=s3cr3t
Đổi config không cần build lại image: sửa ConfigMap → rollout lại Pod → app nhận giá trị mới. Cùng 1 image chạy được mọi môi trường.

Để ý: set env kích hoạt một rolling update — Pod được tạo lại với env mới. Đây là hành vi rollout bạn sẽ học kỹ buổi chiều.
